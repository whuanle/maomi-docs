## 1. 项目目标与边界

### 1.1 目标
- 构建一个“以文件系统为内容源”的文档平台，支持 Markdown/MDX。
- 基于 `docs/` 自动生成 Header 模块导航与 Sidebar 多级目录。
- 支持 `/zh`、`/en` 双语路由，默认中文。
- 支持亮/暗主题、代码高亮、图片放大、复制 Markdown 原文。
- 支持内容搜索，并可通过 MCP 对外提供检索与读取能力。
- 可容器化部署，支持后续扩展业务功能（非纯静态站）。

### 1.2 非目标（当前阶段）
- 不做在线富文本编辑后台。
- 不做权限体系（私有文档鉴权）复杂场景。
- 不做多租户内容隔离。

## 2. 核心需求清单（汇总）

### 2.1 页面与交互
- 全局 Header：左侧模块导航、右侧语言切换 + 主题切换。
- 首页：项目介绍 + 图片说明。
- 模块页：左右布局（左侧 Sidebar，右侧正文）。
- Footer：可配置 GitHub、博客、友链。
- 支持亮/暗模式。

### 2.2 内容与规则
- 内容来源于 `docs/` 文件系统。
- 支持 `md` 与 `mdx`。
- 每个目录支持 `_dir.json` 配置排序与显示。
- 未配置目录排序时，支持文件名前缀序号排序（如 `1.`、`12.`）。
- 显示名称与路由不暴露该序号前缀。

### 2.3 平台能力
- 搜索能力（站内搜索）。
- MCP 协议接入（便于外部 AI/工具检索内容）。
- SEO + GEO 友好。
- Docker 部署。

## 3. 技术选型与关键决策

### 3.1 框架：Next.js（推荐）

**结论**：采用 `Next.js App Router + TypeScript`。  
**理由**：
- 同时具备页面渲染、API、缓存重验证、SEO 支撑能力。
- 支持服务端扩展（Route Handlers、Server Components）。
- 兼顾性能（预渲染/ISR）与后续动态功能扩展。

> 说明：`Node.js` 是运行时，`Next.js` 是更高层全栈框架。这里并非二选一冲突。

### 3.2 UI 与组件
- `Tailwind CSS`：快速构建一致设计系统。
- `shadcn/ui`：基础组件复用与可定制。
- `next-themes`：亮/暗主题切换。

### 3.3 Markdown/MDX 渲染
- `@next/mdx` + `remark` + `rehype`。
- 推荐插件：
  - `remark-gfm`
  - `remark-frontmatter`
  - `rehype-pretty-code`（底层 `shiki`）
  - `rehype-sanitize`

### 3.4 搜索
- 主方案：`Meilisearch`（服务端搜索，易扩展过滤与排序）。
- 轻量备选：`Pagefind`（快速上线）。

### 3.5 MCP
- 独立 `mcp-server` 服务（可与主站同仓库多应用部署）。
- 基于统一索引源实现 `search_docs` / `get_doc` 等工具。

## 4. 总体架构

```text
作者维护 docs/ + config/
        |
        v
内容扫描器（构建时/运行时） ----> 文档清单 manifest（含 sourcePath/publicPath）
        |                               |
        |                               +--> Next.js 页面渲染（Header/Sidebar/Doc）
        |
        +--> 索引构建器 ----------------> 搜索引擎（Meilisearch）
                                            |
                                            +--> /api/search
                                            +--> MCP search_docs
```

组件划分：
- `web`：Next.js 站点（渲染、API、SEO）。
- `search`：Meilisearch（可独立容器）。
- `mcp`：MCP server（复用文档索引/搜索 API）。

## 5. 目录与路由规范

### 5.1 内容目录结构（强约定）

```text
docs/
  <module>/                     # 一级目录 = Header 模块
    module.json                 # 模块元信息
    zh/
      _dir.json                 # 当前目录导航配置（可选）
      1.简介.mdx
      2.快速开始.mdx
      10.安装部署/
        _dir.json
        1.Docker.md
        2.Kubernetes.md
    en/
      _dir.json
      1.Intro.mdx
      2.QuickStart.mdx
```

### 5.2 路由规则
- `/` -> 重定向到 `/zh`
- `/:locale`
- `/:locale/:module`
- `/:locale/:module/*slug`

示例：
- `/zh/getting-started`
- `/zh/getting-started/简介`
- `/zh/getting-started/安装部署/docker`

### 5.3 路由映射原则（关键）
- 文件系统路径用于定位源文件（`sourcePath`）。
- 对外 URL 使用规范化路径（`publicPath`）。
- 系统维护双向映射：
  - `sourcePath -> publicPath`
  - `publicPath -> sourcePath`
- 渲染、搜索跳转、上一篇/下一篇统一使用 `publicPath`。

## 6. 模块与配置规范

### 6.1 全局配置（`config/`）

```text
config/
  site.json
  i18n.json
  theme.json
  search.json
  mcp.json
```

`config/site.json` 示例：

```json
{
  "title": { "zh": "猫咪文档", "en": "Maomi Docs" },
  "description": {
    "zh": "基于 Markdown/MDX 的文档平台",
    "en": "A docs platform powered by Markdown/MDX"
  },
  "defaultLocale": "zh",
  "locales": ["zh", "en"],
  "footerLinks": [
    { "label": "GitHub", "href": "https://github.com/your-org/your-repo" },
    { "label": "Blog", "href": "https://blog.example.com" }
  ]
}
```

### 6.2 模块配置（`docs/<module>/module.json`）

```json
{
  "id": "getting-started",
  "title": { "zh": "快速开始", "en": "Getting Started" },
  "order": 10,
  "hidden": false,
  "defaultDoc": "简介"
}
```

字段说明：
- `order`：Header 模块顺序。
- `hidden`：是否在 Header 中隐藏。
- `defaultDoc`：访问 `/:locale/:module` 时默认打开文档（`publicPath` 相对路径）。

## 7. 目录级配置（`_dir.json`）与排序规则

### 7.1 `_dir.json` 基础结构

```json
{
  "order": ["1.简介.mdx", "10.安装部署", "20.FAQ.mdx"],
  "hidden": ["99.草稿.mdx"],
  "titles": {
    "1.简介.mdx": "快速介绍",
    "10.安装部署": "安装部署"
  },
  "collapsed": ["20.FAQ"]
}
```

### 7.2 可选扩展字段

```json
{
  "expanded": ["10.安装部署"],
  "links": [
    { "label": "外部 API 文档", "href": "https://example.com/api", "order": 90 }
  ],
  "separators": [
    { "id": "basic", "label": "基础", "order": 1 },
    { "id": "advanced", "label": "进阶", "order": 50 }
  ],
  "maxDepth": 4
}
```

### 7.3 作用域与继承
- `_dir.json` 仅作用于当前目录，不递归影响子目录。
- 子目录的 `_dir.json` 独立生效。

### 7.4 同级排序优先级（最终规则）
1. `_dir.json.order`（显式排序）
2. 文件/目录名前缀序号（如 `1.`、`12.`）
3. Frontmatter `order`
4. 目录优先于文件（默认 `true`）
5. 名称字典序

### 7.5 标题显示优先级
1. `_dir.json.titles[item]`
2. Frontmatter `sidebarTitle`
3. Frontmatter `title`
4. 去序号后的文件名/目录名

### 7.6 容错策略
- `_dir.json` 解析失败：降级到默认扫描规则，记录错误日志。
- `order`/`hidden`/`titles` 引用不存在条目：忽略并告警。
- 缺失标题：回退到文件名。

## 8. 文件名前缀序号规则（重点）

### 8.1 目标
为便于作者在文件系统中排序，允许文件名使用序号前缀；但用户界面和 URL 不展示该序号。

### 8.2 匹配模式
- 正则：`^(\d+)\.(.+)$`
- 示例：
  - `1.简介.mdx`
  - `12.安装部署/`
  - `100.API 规范.md`

### 8.3 触发条件
- 当前目录不存在 `_dir.json` 或 `_dir.json` 未配置 `order` 时，前缀序号是主排序依据。
- 若 `order` 存在，前缀序号用于未列入 `order` 的补位条目。

### 8.4 展示规则
- 侧边栏、面包屑、页面标题默认去除序号前缀。
- 如存在 `titles` / Frontmatter 标题覆盖，按标题优先级显示。

### 8.5 路由规则
- URL 段使用去前缀后的名称再 slugify。
- 示例：
  - `1.简介.mdx` -> `简介`
  - `12.安装部署/1.Docker.md` -> `安装部署/docker`

### 8.6 路由冲突处理
- 若同级条目去前缀后路由段冲突，构建阶段报错并终止。
- 允许通过以下方式消解：
  - 修改文件名（去前缀后保持唯一）
  - 指定 Frontmatter `slug`（显式唯一）

### 8.7 映射清单（建议）
构建产物生成 `doc-manifest.json`：

```json
{
  "items": [
    {
      "sourcePath": "docs/getting-started/zh/1.简介.mdx",
      "publicPath": "/zh/getting-started/简介",
      "locale": "zh",
      "module": "getting-started"
    }
  ]
}
```

## 9. 侧边栏行为规范（完整）

### 9.1 节点类型
- `doc`：文档节点
- `dir`：目录节点
- `link`：外部链接节点
- `separator`：分隔节点

### 9.2 交互规则
- 当前文档节点高亮并设置 `aria-current="page"`。
- 当前路径的祖先目录自动展开。
- `collapsed` 指定目录初始折叠；`expanded` 优先级高于 `collapsed`。
- 用户展开状态持久化：`localStorage` 键 `sidebar:state:{locale}:{module}`。
- 首次进入页面自动滚动到激活项附近。

### 9.3 目录落地页
- 目录下若存在 `index.mdx` 或 `README.md`，视为该目录落地页。
- 点击目录标题：
  - 有落地页 -> 跳转到落地页。
  - 无落地页 -> 仅展开/折叠。

### 9.4 性能策略
- 支持至少 4 级目录。
- 单目录条目过多（如 >200）时，启用虚拟滚动或分批渲染。
- 目录树按 `module + locale + contentHash` 缓存。

## 10. Frontmatter 规范（作者协议）

推荐字段：

```yaml
---
title: 快速开始
sidebarTitle: 开始
description: 项目初始化和环境准备
order: 20
slug: quick-start
tags: [入门, 安装]
draft: false
toc: true
updatedAt: 2026-02-11
---
```

字段约定：
- `title`：页面标题。
- `sidebarTitle`：侧边栏标题覆盖。
- `order`：同级排序数值。
- `slug`：路由段覆盖（需同级唯一）。
- `draft`：草稿开关（生产环境默认隐藏）。
- `toc`：是否展示右侧目录锚点。

## 17. 渲染与缓存策略

### 17.1 渲染策略结论
采用 **ISR + 按需重验证**：
- 文档页面默认预渲染，提高性能与 SEO。
- 内容更新后触发 revalidate（路径或标签）。

### 17.2 缓存分层
- 页面缓存：按路由缓存。
- 目录树缓存：按 `module+locale+contentHash`。
- 搜索缓存：短 TTL（如 30~120 秒）。

### 17.3 更新触发
- Git webhook / CI 构建完成后调用刷新端点。

## 21. 部署设计（Docker）

### 21.1 Next.js Dockerfile（推荐）

```dockerfile
FROM node:22-alpine AS base
ENV NEXT_TELEMETRY_DISABLED=1
RUN corepack enable

FROM base AS deps
WORKDIR /app
COPY package.json pnpm-lock.yaml ./
RUN pnpm install --frozen-lockfile

FROM base AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN pnpm build

FROM node:22-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED=1
RUN addgroup -S nodejs && adduser -S nextjs -G nodejs
COPY --from=builder /app/public ./public
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
USER nextjs
EXPOSE 3000
CMD ["node", "server.js"]
```

`next.config` 需开启：

```js
output: 'standalone'
```

### 21.2 docker-compose（示例）

```yaml
services:
  web:
    build: .
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
      - MEILI_HOST=http://search:7700
      - MEILI_API_KEY=${MEILI_API_KEY}
    depends_on:
      - search

  search:
    image: getmeili/meilisearch:latest
    ports:
      - "7700:7700"
    environment:
      - MEILI_MASTER_KEY=${MEILI_API_KEY}
    volumes:
      - meili_data:/meili_data

  mcp:
    build: ./apps/mcp-server
    environment:
      - MEILI_HOST=http://search:7700
      - MEILI_API_KEY=${MEILI_API_KEY}
    depends_on:
      - search

volumes:
  meili_data:
```

- `_dir.json.order` 生效并可覆盖默认排序。
- 无 `order` 时按序号前缀排序。
- UI 展示名称不包含序号前缀。
- URL 不包含序号前缀，且能正确映射回源文件。
- 去前缀后冲突会在构建期报错。

### 23.3 渲染与体验
- md/mdx 正常渲染。
- 代码高亮覆盖 C#/Python/Bash/Shell/Go。
- 图片可点击放大。
- 页面可复制 Markdown 原文。

### 23.4 平台能力
- 搜索能按关键词返回相关文档并跳转。
- MCP 可执行搜索与文档读取。
- Docker 可启动 web（及可选 search/mcp）服务。

### 23.5 SEO/GEO
- 生成 sitemap、robots、canonical、hreflang。
- 文档页输出结构化数据。

## 24. 风险与对策

- **风险：目录命名不规范导致路由冲突**  
  对策：构建期严格校验并阻断发布。

- **风险：MDX 注入风险**  
  对策：限制组件白名单 + sanitize。

- **风险：搜索索引与文档不同步**  
  对策：CI 增量索引 + 定时全量校验任务。

- **风险：多语言内容不齐**  
  对策：缺失语言统一回退提示页并打出缺失报告。

## 附录 B：序号前缀处理伪代码

```ts
const PREFIX_RE = /^(\d+)\.(.+)$/;

function parseName(name: string) {
  const m = name.match(PREFIX_RE);
  if (!m) return { sortNo: undefined, cleanName: name };
  return { sortNo: Number(m[1]), cleanName: m[2].trim() };
}
```
